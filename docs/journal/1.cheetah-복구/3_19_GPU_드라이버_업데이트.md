→ **"의존성 지옥(Dependency Hell)"** — apt 방식 포기, Runfile 방식으로 전환.

### STEP 5 — Runfile 수동 설치

#### 5-1. 사전 준비

```bash
cd ~
sudo apt-get install -y build-essential gcc make libglvnd-dev pkg-config
wget https://us.download.nvidia.com/XFree86/Linux-x86_64/535.154.05/NVIDIA-Linux-x86_64-535.154.05.run
```

#### 5-2. Nouveau 드라이버 비활성화 (필수)

기존 오픈소스 GPU 드라이버인 Nouveau가 GPU를 점유하고 있으면 NVIDIA 드라이버 설치가 차단됨.

```bash
sudo bash -c "cat <<EOF > /etc/modprobe.d/blacklist-nouveau.conf
blacklist nouveau
options nouveau modeset=0
EOF"

sudo update-initramfs -u
sudo reboot
```

재부팅 후 비활성화 확인:

```bash
lsmod | grep nouveau
# 아무것도 출력되지 않아야 성공
```

#### 5-3. 드라이버 설치

```bash
chmod +x NVIDIA-Linux-x86_64-535.154.05.run
sudo sh ./NVIDIA-Linux-x86_64-535.154.05.run --dkms --no-opengl-files
```

**설치 중 선택지:**

|질문|선택|이유|
|---|---|---|
|The distribution-provided pre-install script failed. Continue?|Continue installation|Ubuntu 18.04 구형 스크립트 오류 — 무시 가능|
|Register kernel module sources with DKMS?|Yes|커널 업데이트 시 드라이버 자동 재빌드|
|Install 32-bit compatibility libraries?|No|서버 환경 불필요|
|Run nvidia-xconfig utility?|No|Headless 서버 환경|

```bash
sudo reboot
nvidia-smi
```

**설치 성공 확인:**

```text
Driver Version: 535.154.05   CUDA Version: 12.2
V100-DGXS-32GB × 4 정상 인식
No running processes found
```

### STEP 6 — nvidia-container-toolkit 재설치

드라이버 업그레이드 과정에서 `apt purge nvidia*` 실행으로 기존 `nvidia-container-toolkit`도 함께 삭제됨.

#### 6-1. 저장소 재등록

```bash
curl -fsSL https://nvidia.github.io/libnvidia-container/gpgkey | \
  sudo gpg --dearmor -o /usr/share/keyrings/nvidia-container-toolkit-keyring.gpg

curl -s -L https://nvidia.github.io/libnvidia-container/ubuntu18.04/libnvidia-container.list | \
  sed 's#deb https://#deb [signed-by=/usr/share/keyrings/nvidia-container-toolkit-keyring.gpg] https://#g' | \
  sudo tee /etc/apt/sources.list.d/nvidia-container-toolkit.list

sudo apt-get update
```

#### 6-2. 설치 시도 → 설정 꼬임 발생

`apt-get install` 후 `daemon.json`, `config.toml` 수동 수정 과정에서 설정이 꼬여 Docker가 NVIDIA 런타임을 인식하지 못하는 상태 진입.

주요 시도 및 실패 내역:

|시도|내용|결과|
|---|---|---|
|`daemon.json` 수동 수정|`"default-runtime": "nvidia"` 추가|Docker 재시작 hang|
|`config.toml` 수동 작성|`ldconfig = "@/sbin/ldconfig"` 등 설정|`OCI runtime create failed`|
|심볼릭 링크 수동 생성|`libnvidia-ml.so.1` 링크 여러 경로에 생성|`nvidia-container-cli list` 결과 빈값 유지|
|AppArmor 해제|`sudo aa-complain /usr/bin/nvidia-container-runtime`|변화 없음|

#### 6-3. reinstall 방식으로 해결

설정 수동 수정을 포기하고 기존 설정 파일을 모두 제거한 뒤 재설치.

```bash
# 기존 꼬인 설정 파일 전체 제거
sudo rm -f /etc/docker/daemon.json
sudo rm -f /etc/nvidia-container-runtime/config.toml

# toolkit 재설치
sudo apt-get install --reinstall nvidia-container-toolkit

# nvidia-ctk로 Docker 설정 자동 생성
sudo nvidia-ctk runtime configure --runtime=docker

sudo systemctl daemon-reload
sudo systemctl restart docker
```

**설치 성공 확인:**

```bash
sudo docker info | grep -i runtime
# Runtimes: nvidia runc
# Default Runtime: nvidia
```

### STEP 8 — K8s Device Plugin GPU 인식 확인 → 실패

#### 8-1. 노드 복귀 및 상태 확인

```bash
# 마스터 노드에서 실행
kubectl uncordon v100dgx-gpu04
kubectl describe node v100dgx-gpu04 | grep -A 5 "Allocatable"
```

**결과:**

```text
Allocatable:
  cpu:                40
  memory:             263955020Ki
  nvidia.com/gpu:     0     ← GPU 인식 안 됨
  pods:               110
```

#### 8-2. Device Plugin 로그 확인

```bash
kubectl get pods -n kube-system -o wide | grep nvidia | grep v100dgx-gpu04
kubectl logs [pod-name] -n kube-system
```

**Device Plugin 로그:**

```text
W  No valid resources detected, creating a null CDI handler
E  Detected non-NVML platform: could not load NVML library:
   libnvidia-ml.so.1: cannot open shared object file: No such file or directory
E  Incompatible platform detected
I  No devices found. Waiting indefinitely.
```

#### 8-3. 원인 분석

Docker에서는 `--gpus all` 플래그로 NVIDIA 런타임이 라이브러리를 컨테이너 안으로 직접 주입하기 때문에 GPU 인식이 성공함.

반면 K8s Device Plugin은 실행 시 **호스트의 표준 라이브러리 경로를 직접 참조**해서 GPU를 카운트하는 방식.  
Ubuntu 18.04의 구형 라이브러리 경로 체계 + 최신 toolkit 버전의 경로 탐색 규칙 사이의 불일치로 인해 `libnvidia-ml.so.1`을 찾지 못함.

심볼릭 링크 추가, `config.toml` 경로 수정, Kubelet 재시작, Device Plugin 재기동 등 다수 시도를 반복했으나 모두 실패.

**근본 원인:** Ubuntu 18.04는 EOL 상태로, 최신 nvidia-container-toolkit(1.19.0)이 요구하는 시스템 환경을 더 이상 안정적으로 지원하지 않음.

## 4. 재구축 결정 근거

세 단계를 순서대로 시도하고 직접 한계를 확인한 뒤 결정.

1. `apt` 방식 → Ubuntu 18.04 의존성 충돌로 드라이버 설치 자체 불가
2. Runfile 방식 → 드라이버 설치 성공, nvidia-container-toolkit 설치 성공, Docker GPU 인식 성공
3. K8s 연동 → Device Plugin이 Ubuntu 18.04 라이브러리 경로 체계에서 `libnvidia-ml.so.1` 탐색 실패

**재구축 결정의 핵심은 toolkit 설치 실패가 아니라 K8s Device Plugin의 GPU 인식 실패였다.** Docker 레벨에서는 GPU가 정상 인식됐음에도 불구하고, K8s Device Plugin은 `nvidia.com/gpu: 0`을 유지했다. 심볼릭 링크 경로 4종 추가(`/usr/lib/x86_64-linux-gnu`, `/usr/local/nvidia/lib64` 등), `config.toml`의 `ldconfig` 경로 변경, Kubelet 재시작, Device Plugin DaemonSet 재기동, 호스트 PATH 환경변수 주입 등 가능한 경로를 모두 시도했으나 결과는 동일했고, 이 문제가 Ubuntu 18.04의 라이브러리 경로 체계와 최신 toolkit의 구조적 불일치에서 비롯된 것임을 확인했다.

패치로 해결할 수 있는 설정 문제가 아니라 OS 레벨의 구조적 문제였기 때문에, 이후에도 동일한 문제가 반복될 것이라 판단하고 전면 재구축을 결정했다.

## 7. 핵심 인사이트

이번 작업에서 가장 중요한 판단은 **"언제 포기할 것인가"** 였다.

Docker 레벨에서 GPU가 잡힌다는 것은 드라이버와 하드웨어 자체는 정상이라는 증거였다. 문제는 K8s Device Plugin이 `nvidia.com/gpu: 0`을 유지한 채 해결되지 않는다는 것이었고, 가능한 모든 경로(심볼릭 링크 4종 / config.toml / daemon.json / Kubelet 재시작 / Device Plugin 재기동)를 시도해도 결과가 달라지지 않았다.

패치해야 할 곳이 설정 파일이 아니라 Ubuntu 18.04의 라이브러리 경로 체계 자체라는 것을 확인한 시점에서, 이 문제는 앞으로도 반복될 구조적 문제라고 판단했다. 드라이버도 설치되고 Docker GPU 인식도 됐지만, K8s Device Plugin과의 연동이 근본적으로 불가능한 환경이라는 결론이었다. 이를 직접 검증한 뒤 재구축을 결정한 것은 기술적으로 올바른 판단이었다.
