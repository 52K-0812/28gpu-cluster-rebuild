> **작업 일자:** 2026-03-31
> **대상:** 클러스터 전체 (master-01/02, GPU 워커 노드 4대)

## 3. 원인 분석

### 3-1. 레이어별 진단

| 단계 | 명령어                                                                | 결과                              |
| ---- | --------------------------------------------------------------------- | --------------------------------- |
| 1    | `kubectl get pods -n calico-system`                                   | 전체 Running ✅                   |
| 2    | `kubectl get nodes`                                                   | 전체 Ready ✅                     |
| 3    | 워커노드에서 `curl -k https://10.96.0.1:443/healthz`                  | timeout ❌                        |
| 4    | 워커노드에서 `curl -k https://LB_PUBLIC_IP:6443/healthz` | timeout ❌                        |
| 5    | 워커노드에서 `ping LB_PUBLIC_IP`                         | 정상 ✅                           |
| 6    | master-01에서 `ping LB_PUBLIC_IP~156`                    | 152, 153, 155, 156 → 100% loss ❌ |
| 7    | master-01에서 tcpdump                                                 | SYN만 있고 SYN-ACK 수신 없음 ❌   |
| 8    | v100에서 tcpdump                                                      | SYN-ACK 정상 송신 ✅              |
| 9    | `sudo conntrack -L \| grep UNREPLIED \| wc -l`                        | 20개 이상 ❌                      |

### 3-2. 원인 1 — conntrack 테이블 오염

Prometheus가 master-01에서 실행되며 모든 워커노드 10250 포트를 주기적으로 scrape. 이 과정에서 conntrack NAT 포트 매핑이 꼬임.

**오염된 엔트리 패턴:**

```
tcp  SYN_SENT src=192.168.184.95 dst=LB_PUBLIC_IP sport=54782 dport=10250 [UNREPLIED]
              src=LB_PUBLIC_IP  dst=LB_PUBLIC_IP sport=10250 dport=13984
```

요청 포트(54782)와 conntrack이 기대하는 응답 포트(13984) 불일치 → SYN-ACK 드롭.

### 3-3. 원인 2 — 스위치 비대칭 ACL (Port Isolation)

master-01 재부팅 후에도 재발. ping 테스트로 패턴 확인:

| 노드          | IP                        | master-01에서 ping |
| ------------- | ------------------------- | ------------------ |
| master-02     | LB_PUBLIC_IP | ❌ 100% loss       |
| v100-gpu-01   | LB_PUBLIC_IP | ❌ 100% loss       |
| 2080ti-gpu-02 | LB_PUBLIC_IP | ✅ 정상            |
| 2080ti-gpu-03 | LB_PUBLIC_IP | ❌ 100% loss       |
| 2080ti-gpu-04 | LB_PUBLIC_IP | ❌ 100% loss       |
| NAS           | LB_PUBLIC_IP | ✅ 정상            |

ARP 테이블엔 MAC 있지만 실제 패킷 전달 안 됨 → 스위치 Port Isolation.

### 3-4. 원인 3 — ClusterIP 라우팅 누락

```bash
# 워커노드에서
ip route get 10.96.0.1
# 결과: 10.96.0.1 via LB_PUBLIC_IP dev eno5  ← 외부 게이트웨이로 나가버림
```

Kubernetes ClusterIP 대역(`10.96.0.0/12`)이 게이트웨이로 라우팅 → 학교 ACL에서 차단. master-01은 API 서버 자체라 로컬 처리되지만, 워커노드들은 외부로 나가서 막힘.

### 3-5. 장애 전파 경로

```
Prometheus(master-01) → conntrack 오염
         +
스위치 Port Isolation (특정 노드 간 L2 차단)
         +
ClusterIP 10.96.0.0/12 라우팅 누락 (게이트웨이로 나가서 ACL 차단)
    ↓
master-01 ↔ 워커노드 kubelet(10250) 통신 불가
    ↓
Calico CNI → API 서버(10.96.0.1:443) TLS 연결 실패
    ↓
신규 Pod sandbox 생성 불가 (JupyterHub, 시스템 파드 전체)
```

## 4. 해결 과정

### 4-1. conntrack 초기화 (임시 완화)

```bash
sudo conntrack -F
sudo reboot
```

재부팅 후 일시 정상화 → 스위치 ACL 문제로 재발 확인.

### 4-2. Prometheus master-02 강제 배치 (conntrack 오염 근본 해결)

Prometheus가 master-01에 있는 한 conntrack 오염 재발. Prometheus CR 직접 수정.

```bash
kubectl edit prometheus monitoring-kube-prometheus-prometheus -n monitoring
# spec: 아래에 추가
# nodeSelector:
#   kubernetes.io/hostname: master-02

kubectl delete pod prometheus-monitoring-kube-prometheus-prometheus-0 -n monitoring
```

> **주의:** StatefulSet patch나 Helm upgrade로는 Prometheus Operator가 계속 덮어씀. 반드시 Prometheus CR을 직접 수정해야 함.

### 4-3. 시스템 파드 전체 master-02 고정 배치

GPU 노드를 학습 전용으로 비우고, 시스템 파드 전체를 master-02로 이전.

```bash
# Deployment 패치
for item in "ai-team hub" "ai-team proxy" "monitoring monitoring-kube-prometheus-operator" "portainer portainer"; do
  ns=$(echo $item | awk '{print $1}')
  name=$(echo $item | awk '{print $2}')
  kubectl patch deployment $name -n $ns --type=json \
    -p='[{"op":"add","path":"/spec/template/spec/nodeSelector","value":{"kubernetes.io/hostname":"master-02"}}]'
done

# StatefulSet 패치
kubectl patch statefulset alertmanager-monitoring-kube-prometheus-alertmanager -n monitoring --type=json \
  -p='[{"op":"add","path":"/spec/template/spec/nodeSelector","value":{"kubernetes.io/hostname":"master-02"}}]'
```

> **주의:** Helm upgrade 시 nodeSelector 날아갈 수 있음. 업그레이드 시 values.yaml에 추가 필요.

### 4-4. Grafana PVC 연결 수정

`storage` 볼륨이 `emptyDir`로 되어있어 재시작 시 설정 초기화되는 문제 발견 및 수정.

```bash
kubectl patch deployment monitoring-grafana -n monitoring --type=json -p='[
  {
    "op": "replace",
    "path": "/spec/template/spec/volumes/1",
    "value": {
      "name": "storage",
      "persistentVolumeClaim": {"claimName": "grafana-pvc"}
    }
  }
]'
```

### 4-5. master-01 NoSchedule taint 적용 (구조적 개선)

master-01을 컨트롤 플레인 전용으로 보호. 이후 신규 파드는 master-01에 스케줄링되지 않음.

```bash
kubectl taint node master-01 node-role.kubernetes.io/control-plane:NoSchedule
```

### 4-6. ClusterIP 라우팅 영구 설정 (master-02 적용 완료)

```bash
sudo tee /etc/netplan/99-k8s-routes.yaml <<EOF
network:
  version: 2
  ethernets:
    eno5:
      routes:
        - to: 10.96.0.0/12
          via: LB_PUBLIC_IP
EOF
sudo netplan apply

# 확인
curl -k --connect-timeout 5 https://10.96.0.1:443/healthz
# 결과: ok ✅
```

> **TODO:** 나머지 워커노드(v100-gpu-01, 2080ti-gpu-02~04)에도 동일 적용 필요. 인터페이스명 확인 후 적용 (`eno5` 또는 `enp2s0f0` 등 노드마다 다를 수 있음).

### 4-7. 스위치 ACL 수정 요청 (진행 중)

네트워크 담당자에게 요청:

> LB_PUBLIC_IP ~ LB_PUBLIC_IP 구간 서버들 간 포트 격리(Port Isolation)가 설정되어 일부 서버 간 직접 통신이 차단되고 있습니다. 해당 구간 전체 L2 통신 허용 요청드립니다.

## 7. 핵심 인사이트

- **Pod Running, Calico Running인데 TLS timeout → conntrack 먼저 확인** `sudo conntrack -L | grep UNREPLIED | wc -l` — 20개 이상이면 오염 의심
- **conntrack -F 후 재발 → Prometheus 위치가 근본 원인** 재부팅이 아니라 Prometheus를 다른 노드로 옮기는 게 해결책
- **master-01에서만 안 되고 다른 노드에서 된다 → 스위치 ACL 비대칭 의심** tcpdump로 양쪽에서 패킷 흐름 확인 필수
- **ping 테스트로 스위치 격리 범위 파악** 어떤 노드끼리 되고 안 되는지 패턴 보면 포트 그룹 문제 바로 확인 가능
- **ClusterIP 라우팅 누락은 워커노드 공통 문제** 신규 노드 추가 시 `99-k8s-routes.yaml` 배포를 표준 절차에 포함
- **Prometheus Operator는 StatefulSet을 계속 reconcile함** StatefulSet patch가 아니라 Prometheus CR 직접 수정 또는 Helm values에 명시해야 nodeSelector 유지
- **alertmanager도 동일 원칙 적용** `--set alertmanager.alertmanagerSpec.nodeSelector` 로 Helm values에 명시해야 영구 반영
- **Grafana 롤링업데이트 중 파드 2개 공존은 정상** 신규 파드가 CrashLoopBackOff면 강제 삭제로 정리 가능 (stateless라 안전)
- **설계 원칙(SoC)을 초기에 강제하지 않으면 운영 중 장애로 돌아온다** master-01 taint는 클러스터 초기화 시점에 바로 적용했어야 함

```bash
# Pod 생성 실패 시 이벤트 확인
kubectl describe pod [pod-name] -n [namespace] | tail -20

# Calico TLS timeout → 워커노드에서 ClusterIP 접근 테스트

# conntrack 오염 빠른 확인

# 특정 노드 10250 오염 확인
sudo conntrack -L | grep "LB_PUBLIC_IP.*10250" | grep UNREPLIED

# 패킷 흐름 확인 (양방향)
sudo tcpdump -i eno5 host LB_PUBLIC_IP and port 10250 -n -c 10   # master-01에서
sudo tcpdump -i enp2s0f0 port 10250 -n -c 10                       # v100에서

# conntrack 초기화 (임시)
sudo conntrack -F

# 스위치 격리 범위 확인
for ip in LB_PUBLIC_IP LB_PUBLIC_IP LB_PUBLIC_IP LB_PUBLIC_IP LB_PUBLIC_IP LB_PUBLIC_IP; do
  result=$(ping -c 1 -W 1 $ip > /dev/null 2>&1 && echo "✅" || echo "❌")
  echo "$ip $result"
done

# ClusterIP 라우팅 확인
ip route get 10.96.0.1

# Prometheus 위치 확인 (master-02여야 함)
kubectl get pod -n monitoring -o wide | grep prometheus-monitoring-kube-prometheus-prometheus-0

# master-01 taint 확인
kubectl describe node master-01 | grep Taints
```

## 9. 후속 작업 — alertmanager nodeSelector 고정 (2026-03-31 오후)

### 9-1. 문제 현상

`helm upgrade` 이후 alertmanager가 `2080ti-gpu-03`에 배치됨. StatefulSet 직접 patch 시도했으나 Prometheus Operator가 계속 reconcile하여 nodeSelector가 반영되지 않음.

```bash
# patch 후에도 nodeSelector 없음 확인
kubectl get statefulset alertmanager-monitoring-kube-prometheus-alertmanager \
  -n monitoring -o yaml | grep -A5 nodeSelector
# 결과: 아무것도 안 뜸 → patch 무효
```

### 9-2. 원인

Prometheus Operator가 관리하는 StatefulSet은 직접 patch해도 Operator가 덮어씀. Helm values에 명시해야 영구 반영됨.

### 9-3. 해결 — Helm values에 nodeSelector 추가

```bash
helm upgrade monitoring prometheus-community/kube-prometheus-stack \
  -n monitoring \
  --reuse-values \
  --set alertmanager.alertmanagerSpec.nodeSelector."kubernetes\.io/hostname"=master-02 \
  --timeout 120s
```

### 9-4. Grafana 롤링업데이트 파드 충돌

helm upgrade 과정에서 Grafana 파드 2개 동시 존재 발생:

```
monitoring-grafana-6dfdf7f968-vnrvt   3/3  Running          ← 기존 파드
monitoring-grafana-785895c69d-8ddnp   2/3  CrashLoopBackOff ← 신규 파드
```

신규 파드의 sidecar가 Grafana 메인 프로세스 기동 전에 reload 시도 → 일시적 CrashLoopBackOff. 롤링업데이트가 멈춰있어 강제 삭제로 정리:

```bash
kubectl delete pod monitoring-grafana-785895c69d-8ddnp -n monitoring --force
```

> **참고:** Grafana는 stateless라 강제 삭제 안전. etcd/DB 등 stateful 파드는 강제 삭제 시 데이터 손상 위험.

### 9-5. 최종 결과

```

grafana      → master-02 ✅ Running (단일 파드)
prometheus   → master-02 ✅ Running
```
