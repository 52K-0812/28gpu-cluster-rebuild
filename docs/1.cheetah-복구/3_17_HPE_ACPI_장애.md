# 🛡️ [트러블슈팅] ## HPE ACPI 로그 스팸 장애 및 콘솔 화면 변색

---

## 1. 개요

| 항목          | 내용                                                                           |
| ------------- | ------------------------------------------------------------------------------ |
| **목적**      | 콘솔을 도배하는 ACPI 에러 로그 스팸 차단 및 화면 변색 영구 해결                |
| **대상**      | HPE ProLiant (Cheetah-app 노드), Ubuntu 18.04.5 LTS, Kernel 4.15.0-112-generic |
| **핵심 전략** | GRUB 커널 파라미터 튜닝으로 소프트웨어 레벨에서 영구 억제                      |

---

## 2. 문제 현상

- 서버 터미널 및 물리 콘솔에 ACPI 에러 메시지가 초당 수회씩 반복 출력
- 콘솔 화면이 노란색(Amber)으로 변색

**반복 출력 에러:**

```
ACPI Error: SMBus/IPMI/GenericSerialBus write requires Buffer of length 66, found length 32
ACPI Exception: AE_AML_BUFFER_LIMIT, Evaluating _PMM
```

## 3. 원인 분석

| 원인          | 내용                                                                                                                                  |
| ------------- | ------------------------------------------------------------------------------------------------------------------------------------- |
| **로그 스팸** | HPE iLO(전력 관리 칩셋)가 `_PMM` 데이터 전송 시 커널 ACPI 드라이버가 예상하는 버퍼 크기(66)와 실제 데이터(32) 불일치 → 예외 반복 발생 |
| **화면 변색** | HPE 하드웨어 상태 알림 기능이 커널 `splash` 옵션과 간섭 → 콘솔 색상 프로필 왜곡                                                       |
| **분류**      | 구형 커널(4.15)과 HPE 펌웨어 간 호환성 문제. 시스템 운영 및 GPU 연산에는 지장 없는 로그 스팸으로 판명                                 |

---

## 4. 해결 과정

### 4.1 시도 이력

| 시도     | 조치                                             | 결과                                               |
| -------- | ------------------------------------------------ | -------------------------------------------------- |
| 1차      | `GRUB_CMDLINE_LINUX_DEFAULT`에 `loglevel=3` 적용 | 실패 — Critical 등급으로 인식되어 계속 출력        |
| 2차      | `sudo dmesg -n 1` 로그 레벨 하향                 | 실패 — 실시간 터미널 출력 억제 불가                |
| 3차      | `sudo dmesg -D` 콘솔 로그 전체 차단              | 임시 성공 — 현재 세션에서만 유효, 재부팅 시 초기화 |
| 4차      | `loglevel=0`, `splash` 등 적용                   | 부분 성공 — 로그는 멈췄으나 화면 노랗게 변색       |
| **최종** | `splash` 제거 + 시스템 로그 레벨 강제 하향       | **성공** — 로그 스팸 중단 및 화면 색상 정상 복구   |

---

### 4.2 GRUB 커널 파라미터 수정 (영구 해결)

**🛠️ 사용 명령어:**

```bash
sudo nano /etc/default/grub
```

```bash
# 수정 전
GRUB_CMDLINE_LINUX_DEFAULT="quiet splash"

# 수정 후
GRUB_CMDLINE_LINUX_DEFAULT="quiet loglevel=0 acpi_enforce_resources=lax rd.udev.log_priority=3 systemd.show_status=false"
```

> 주의: `splash` 옵션을 반드시 제거해야 HPE 서버 화면 변색 문제가 해결된다.

**파라미터 설명:**

| 파라미터                     | 역할                                     |
| ---------------------------- | ---------------------------------------- |
| `loglevel=0`                 | 시스템 정지 수준 외 모든 커널 로그 차단  |
| `acpi_enforce_resources=lax` | 커널-하드웨어 자원 충돌 시 유연하게 처리 |
| `rd.udev.log_priority=3`     | udev 장치 관리자 로그를 에러 단계로 제한 |
| `systemd.show_status=false`  | 부팅 시 systemd 서비스 상태 메시지 억제  |

---

### 4.3 설정 반영 및 재부팅

**🛠️ 사용 명령어:**

```bash
sudo update-grub
sudo reboot
```

## 5. 결과

- ACPI 에러 로그 스팸 완전 차단, 콘솔 가독성 확보
- 화면 노란색 변색 현상 해소, 정상 색상 복구
- Cheetah 서비스 및 GPU 노드(2080Ti/V100) 전체 정상 운영 확인

## 6. 핵심 인사이트

- **하드웨어 고장과 호환성 문제 구분** — 에러 메시지가 반복 출력되더라도 실제 시스템 영향을 먼저 파악해야 한다. 본 장애는 운영에 무해한 로그 스팸이었고, 포맷이나 재설치 없이 커널 파라미터 튜닝만으로 해결됐다.
- **단계적 시도 기록** — 4번의 실패 시도를 기록해둔 덕분에 무엇이 안 되는지 명확히 알고 최종 해결책에 도달할 수 있었다.
- **추후 계획** — HPE 펌웨어(SPP) 업데이트 및 커널 버전 업그레이드를 통해 근본적인 드라이버 호환성 해결 검토 예정.
